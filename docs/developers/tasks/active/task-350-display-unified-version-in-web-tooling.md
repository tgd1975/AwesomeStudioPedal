## Description

Once TASK-260 lands the unified version number across all manifests, the three
web surfaces need to surface that number to the user:

- [docs/simulator/index.html](docs/simulator/index.html) — web simulator
- [docs/tools/config-builder/index.html](docs/tools/config-builder/index.html) — profile builder
- [docs/tools/configuration-builder/index.html](docs/tools/configuration-builder/index.html) — configuration builder

Today none of them display a version, so a user reporting "the simulator is
broken" can't tell us which deployed build they hit (GitHub Pages caches
across releases). Add a small, unobtrusive `vX.Y.Z` stamp in a consistent
location across all three surfaces.

## Acceptance Criteria

- [ ] All three surfaces render the unified version string in a footer or header corner — same visual treatment across the three for consistency.
- [ ] Version is sourced from a single shared file (e.g. `docs/tools/version.js` or a generated `VERSION.txt` fetched at runtime) — not hard-coded per-surface, so a release bump doesn't require touching three HTML files.
- [ ] Manual verification: open each of the three pages, confirm the same `vX.Y.Z` is displayed.

## Test Plan

No automated tests required — change is non-functional (static markup + a
small fetch). The shared version source is regenerated by the `/release`
skill (extended in TASK-260).

Manual: open all three pages locally (or on the deployed Pages site after a
release) and confirm the version stamp matches `include/version.h`.

## Notes

- Simplest implementation: a `version.js` (or `version.json`) at a known path containing the canonical version, fetched by each page on load and injected into a `<span class="version-stamp">`.
- Keep the three styles consistent: same font size, same corner placement, same colour. The logo branding tasks (TASK-339/340/341) already establish a visual pattern across these three surfaces — match that.
- The `/release` skill (after TASK-260) must regenerate the shared version source on bump, otherwise the displayed number drifts on every release.
- TASK-260 is a **hard prerequisite**: don't display a version that's still drifted across manifests.
