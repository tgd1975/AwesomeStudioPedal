## Description

When a release is cut, the live `docs/developers/tasks/OVERVIEW.md`,
`EPICS.md`, and `KANBAN.md` keep evolving — by the time someone looks
back at `archive/v0.3.0/` six months later, the live files describe a
much-changed world. The per-release `archive/<version>/OVERVIEW.md`
generated today only lists the closed tasks for that version; it does
not preserve what the project's overall task / epic / kanban state
*looked like* at the moment of the release.

Fix: snapshot the three top-level overviews into the per-release
`archive/<version>/` folder during the release flow, renamed with the
version as a suffix:

- `OVERVIEW_<version>.md`
- `EPICS_<version>.md`
- `KANBAN_<version>.md`

The snapshots are **static** — they are demoted from auto-generated
documents to frozen artifacts. All "auto-generated by housekeep" /
`<!-- GENERATED -->` / `<!-- BURNUP:START -->` markers and
"do not edit manually" notices are stripped on the way in, so the
file reads as a one-shot historical document and `housekeep.py` will
not touch it on later runs.

The existing `archive/<version>/OVERVIEW.md` (the per-release "tasks
closed in this version" listing generated by `update_task_overview.py`)
is **not** changed — that file keeps its current name and content.
The new snapshots live next to it under their suffixed names.

## Snapshot timing

The snapshot freezes the **post-archive** state — closed tasks have
already been moved into `archive/<version>/`, OVERVIEW / EPICS /
KANBAN have been regenerated against that state, and the result
reflects what shipped. Concretely, this fits between the existing
release-skill steps:

1. Move closed tasks into `archive/<version>/` (existing step 6).
2. Regenerate overviews via `housekeep.py --apply` (existing step 7).
3. **New:** copy the three live overviews into
   `archive/<version>/{OVERVIEW,EPICS,KANBAN}_<version>.md`,
   stripping the auto-generation markers and notices on the way.
4. CHANGELOG / README / tag / push (existing steps 8+).

## Notes

- The release flow is documented in
  [.claude/skills/release/SKILL.md](.claude/skills/release/SKILL.md).
  The new snapshot step inserts between the existing housekeep step
  and the CHANGELOG step.
- "Stripping markers" means deleting the marker lines themselves
  *and* the auto-generated boilerplate they wrap (e.g. the
  `<!-- BURNUP:START -->` … `<!-- BURNUP:END -->` block stays as
  rendered Markdown, just without the comment delimiters and any
  "auto-generated" disclaimer above it). The chart bodies remain —
  they are valid Markdown / Mermaid and are what the reader is
  actually here for.
- Filename collision with the existing `archive/<version>/OVERVIEW.md`
  is avoided by the `_<version>` suffix on every snapshot. Consider
  whether to also rename the legacy per-release closed-tasks file —
  decision: **no**, leave it alone (out of scope; renaming it would
  break older intra-repo links).
